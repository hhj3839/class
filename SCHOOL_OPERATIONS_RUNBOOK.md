# 우리반 이음 학교 운영·장애 대응 절차

> 대상: 담임교사와 학교 정보 담당자  
> 원칙: 학생 이름, 응답 원문, 참여 링크, 백업 파일을 메신저·공개 문서·GitHub 이슈에 올리지 않는다.

## 1. 수업 전 10분 점검

1. 배포 사이트와 학생 참여 링크를 각각 한 번 연다.
2. 담임 계정 로그인과 담당 학급 이름을 확인한다.
3. 참여 링크가 `?join=` 토큰을 사용하고 QR과 같은 주소인지 확인한다.
4. 설문 관리에서 이번 달과 현재 명단을 확인한다.
5. 운영 학급 JSON 백업을 학교가 승인한 암호화 저장공간에 보관한다.
6. 시험 중단 기준과 종이 대체 활동을 진행 교사와 공유한다.

## 2. 장애별 즉시 대응

| 상황 | 교사가 먼저 할 일 | 재개 조건 |
|---|---|---|
| 학생 화면이 열리지 않음 | QR 대신 참여 링크를 직접 입력하고 같은 네트워크의 다른 기기 1대로 비교 | 두 기기에서 명단 표시 |
| Wi-Fi 끊김 | 제출을 반복하지 말고 연결 복구 후 화면 안내를 확인 | 임시 작성 내용 유지와 명단 재조회 |
| 제출 결과 불명확 | 학생은 같은 탭을 유지하고 담임은 제출 현황을 새로고침 | 완료 안내와 응답 1건 확인 |
| 교사 로그인 만료 | 새로고침 후 다시 로그인하고 학급 이름을 재확인 | 이전 학급 자료 미노출 |
| AI 분석 실패 | 저장된 결과를 사용하고 설문·학생 기록 기능을 계속 운영 | 기본 기록 기능 정상 |
| PDF 실패 | 브라우저 다운로드 허용과 저장 공간을 확인하고 다른 교사용 기기에서 1회 재시험 | 한글과 페이지 확인 |

## 3. 즉시 중단하고 P0로 기록할 상황

- 다른 교사나 로그아웃 화면에 학생 자료가 보임
- 완료 안내가 나왔지만 응답이 저장되지 않음
- 한 번의 제출로 중복 행이 생성됨
- 운영 학급 자료가 복구 시험 중 변경됨
- `바로 도와주세요` 응답이 교사 화면에 반영되지 않음

이 경우 추가 시험을 멈추고 참여 링크 공유를 중단한다. 학생에게는 설문 내용을 다시 공개적으로 말하게 하지 않고 담임이 별도로 확인한다.

## 4. 백업과 복구

### 백업

1. 학급 설정의 개인정보 운영 화면에서 `백업 내보내기`를 실행한다.
2. 파일에 학생 이름과 응답 원문이 포함된다는 경고를 확인한다.
3. 파일을 학교 승인 저장공간으로 옮기고 개인 다운로드 폴더의 복사본을 정리한다.
4. 생성 시각, 실행자와 보관 위치만 운영 기록에 남긴다.

### 복구 시험

1. 운영 학급이 아닌 별도 시험 학급을 준비한다.
2. 시험 학급에서만 `백업 복구`를 실행한다.
3. 학생·응답 UUID가 새로 생성되고 운영 학급 건수가 바뀌지 않았는지 확인한다.
4. 실패하면 반복 실행하지 말고 오류 문구에서 개인정보를 제거해 기록한다.

## 5. 장애 보고에 포함할 정보

- 점검표 ID, 발생 시각, 기기 종류, 운영체제, 브라우저 버전
- 화면 방향과 네트워크 상태
- 배포 커밋, 기대 결과, 실제 결과와 재현 횟수
- 응답 유실·중복 여부와 임시 대응

학생 이름, 응답 내용, 교사 이메일, 학급 ID, 참여 토큰과 개인정보가 보이는 화면 캡처는 제외한다. 기록 양식은 `FIELD_ISSUE_TEMPLATE.md`를 사용한다.

## 6. 수업 종료

1. 담임 화면에서 참여 인원과 즉시 확인할 응답을 확인한다.
2. 다운로드한 PDF와 백업 파일의 보관 위치를 정리한다.
3. 공용 기기에서 로그아웃하고 뒤로 가기·새로고침으로 학생 자료가 남지 않는지 확인한다.
4. 시험 결과를 `FIELD_PILOT_RESULTS_v1.2.1.md`에 비식별로 기록한다.
