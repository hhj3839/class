# Repository workflow

코드나 문서를 변경하는 모든 작업은 다음 절차까지 완료한다.

1. 관련 테스트를 실행한다.
2. 변경 파일만 명시적으로 스테이징한다.
3. 의미가 드러나는 메시지로 커밋한다.
4. 작업 브랜치를 GitHub에 푸시한다.
5. Pull Request를 생성한다.
6. 테스트와 병합 가능 상태를 확인한다.
7. 이상이 없으면 PR을 `main` 브랜치에 병합한다.
8. GitHub Pages 배포 완료를 확인한다.
9. 배포된 사이트에서 변경 사항을 검증한다.

로컬 변경 상태로만 작업을 종료하지 않는다.

테스트 실패, 병합 충돌, 배포 실패가 있으면 임의로 우회하지 말고 원인을 보고한다.

Supabase DB 마이그레이션이나 Edge Function처럼 GitHub Pages 배포에 포함되지 않는 변경은 별도로 배포하고 결과를 확인한다. 외부 서비스 권한이나 인증이 없어 직접 적용할 수 없다면 필요한 수동 작업을 명확히 안내한다.
