# 우리반 이음 v1.2.1 현장 파일럿 점검표

- 기준 버전: `v1.2.1-prepilot`
- 사전 동결 기준선: `b0c8ce0` (PR #75, 2026-07-23)
- 기준 커밋: 시험 시작 직전 `main`의 최신 배포 커밋을 결과지에 기록
- 대상 시기: 2026년 8월 중순 이후
- 배포 주소: https://hhj3839.github.io/class/

## 1. 시험 원칙

- 실제 학생 응답은 담당 담임만 확인하고 화면 사진·오류 기록에는 이름과 원문을 남기지 않는다.
- 오류 재현에는 `태블릿 A`, `학생-테스트1`처럼 비식별 표기를 사용한다.
- 즉시 도움 요청이 포함된 실제 응답은 시험 자료로 복제하지 않고 학교의 기존 보호 절차에 따라 처리한다.
- 시험 전 백업을 만들고 시험용 제출과 운영 제출을 구분한다.

### 현장 시험 전 변경 관리

- 2026년 7월 23일부터 현장 시험 종료까지 신규 기능과 정보 구조 변경은 보류한다.
- 학생 자료 노출, 제출 유실·중복, 로그인·권한, AI 실행 실패, PDF 생성 실패처럼 현장 시험을 막는 P0·P1 오류만 수정한다.
- 수정이 필요하면 별도 PR, 전체 자동 테스트, GitHub Pages 배포 확인을 거친 뒤 결과지의 기준 커밋을 새 값으로 바꾼다.
- 문구·색상·배치 개선은 시험 결과에 근거가 있을 때만 다음 RC에 반영한다.

## 2. 준비물과 역할

- [ ] 학생용 태블릿 3~5대
- [ ] 담임교사용 PC 또는 태블릿 1대
- [ ] 인쇄 가능한 교사용 기기 1대
- [ ] 안정적인 Wi-Fi와 연결 중단 시험이 가능한 기기 1대
- [ ] 유효한 학생 참여 QR 또는 링크
- [ ] [학생 설문 참여 안내](STUDENT_SURVEY_GUIDE.md)와 [담임 운영 가이드](TEACHER_QUICK_GUIDE.md)
- [ ] 실제 시험 전 [기능 점검용 데모 학급](DEMO_CLASS_GUIDE.md)으로 주요 교사 화면을 사전 확인
- [ ] 담당 담임 계정과 접근 권한이 없는 다른 교사 시험 계정
- [ ] 시험 전 Supabase JSON 백업
- [ ] [오류 기록 양식](FIELD_ISSUE_TEMPLATE.md)

역할은 진행 교사 1명, 학생 역할 3~5명, 결과 기록자 1명으로 나누는 것을 권장한다.

## 3. 실행 순서와 기록 방법

1. 데모 학급에서 `D-01~D-06` 사전 리허설을 완료한다.
2. 시험 직전 배포 커밋과 자동 테스트 통과 여부를 결과지에 기록한다.
3. `S-01~S-09`를 같은 시험 회차에서 순서대로 실행한다.
4. 제출 직후 `T-01~T-10`으로 교사 화면 반영과 AI 월 선택을 확인한다.
5. `P-01~P-06`을 저장 파일과 실제 A4 인쇄물로 확인한다.
6. `A-01~A-04`를 마지막에 실행하고 운영 학급 데이터가 바뀌지 않았는지 확인한다.

- 각 항목은 `통과 / 실패 / 미실행` 중 하나로 기록하며 빈칸으로 두지 않는다.
- 기대 결과를 모두 만족해야 통과다. 일부만 만족하면 실패로 기록한다.
- 실패 즉시 [현장 오류 기록 양식](FIELD_ISSUE_TEMPLATE.md)을 만들고 해당 ID를 결과지에 연결한다.
- P0가 발생하면 추가 시험을 중단한다. P1은 핵심 흐름을 우회할 수 있을 때만 기록 후 계속한다.
- 시간 측정 항목은 시작과 완료 시각을 초 단위로 기록한다.

## 10. 중단 기준

다음 중 하나가 발생하면 추가 시험을 중단하고 P0 오류로 기록한다.

- 다른 교사나 로그인 전 화면에 학생 개인정보가 노출됨
- 제출 완료 안내가 나왔지만 응답이 저장되지 않음
- 한 번의 제출로 중복 행이 생성됨
- 실제 운영 학급 자료가 시험 복구 과정에서 변경됨
- 즉시 도움 요청이 교사 화면에 반영되지 않음
